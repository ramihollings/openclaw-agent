Safe-Config: 39 Security Roles for Autonomous Agents

By agent-core • March 20, 2026 • 12 min read

Solo developers and enterprise engineering teams alike are spending entire weekends wrestling with setup friction. Deploying autonomous agents locally is incredibly powerful, unlocking massive productivity gains, but it introduces a massive risk vector: "accidental mass-deletions," infinite looping, and rogue command execution. The solution isn't to throttle the agent's intelligence—it's to sandbox its execution environment using deterministic, role-based tool locking. Welcome to the era of Safe-Config.

The Vulnerability of Unrestricted Compute

When an artificial intelligence agent is initialized with an elevated shell (like bash or zsh), it inherently possesses the same privileges as the host user. A single hallucination, a misunderstood prompt, or an edge-case prompt injection attack from an external webpage can trigger a destructive command. We've seen instances where an agent, attempting to "clean a directory," executed an rm -rf / equivalent across a mounted volume.

Legacy configurations relied on manual human-in-the-loop (HITL) approvals for every single action. While this prevents destruction, it completely cripples the autonomy that makes the agent valuable in the first place. You don't hire an assistant to ask you for permission to click every button on a keyboard; you hire them to get the job done within a predefined set of boundaries.

The Architecture of Safe-Config

We built the Safe-Config Generator to solve this exact problem. Instead of forcing developers to write custom JSON configurations manually—which is prone to human error—our tool generates strict, mathematically verifiable, role-based config.json payloads in seconds. We have meticulously defined 39 distinct security profiles, ranging from hyper-restricted analytical modes to full-scale, heavily audited development environments.

Deep Dive: The "Researcher" Role

Let's examine how a restricted role operates under the hood. The "Researcher" role is designed for deep intelligence gathering without the ability to modify the host system.

{
  "role": "researcher",
  "permissions": {
    "tools": {
      "allow": ["web_search", "web_fetch", "pdf", "read", "memory_search"],
      "deny": ["exec", "write", "edit", "browser", "canvas", "sessions_spawn"]
    }
  },
  "compaction": {
    "reserveTokensFloor": 100000
  }
}

By explicitly denying write operations (write, edit) and subagent spawning (sessions_spawn), the Researcher role guarantees that your local filesystem remains pristine. It can read massive PDF files, search the web, and grep your local memory, but it physically cannot alter a single byte of code. This is enforced at the API layer; if the LLM attempts to output a tool call for exec, the OpenClaw middleware intercepts the request and returns a deterministic 403 Forbidden error back to the model, forcing it to re-evaluate its approach.

The 39 Profiles: A Spectrum of Trust

Our 39 profiles are broken down into four primary tiers of trust:

  • Tier 1: Read-Only Observers. Roles like the Vault Auditor and Compliance Officer. These agents can only use memory_search and sessions_list. They are completely sandboxed from the internet and the host machine's execution layer.
  • Tier 2: Information Gatherers. Roles like the OSINT Stalker and Data Journalist. They have access to Brave Search MCPs and web scraping utilities but lack filesystem write access.
  • Tier 3: Contained Operators. Roles like the UI Vibe Coder. They can write to specific directories and read files, but they are denied access to the global exec command, meaning they cannot run shell scripts or install unverified npm packages.
  • Tier 4: Super-Users with Guardrails. Roles like the Developer. They have allow: ["*"], but highly destructive tools like exec and sessions_spawn are moved to a confirm: [] array, meaning they operate freely until they hit a critical system boundary, at which point an approval prompt is routed to the human.

Cryptographic Verification via Cloud-Gate

Security is only as good as its weakest link. If a local configuration file can be easily edited by a rogue process, the guardrails fail. To prevent this, mr.technology implements a "Cloud-Gate" API architecture.

When you purchase the Founder License, you are issued a unique, cryptographically secure key (e.g., FOUNDER-A1B2C3D4). When your local agent boots, it validates this key against our backend. The backend then streams the immutable JSON configuration directly into the agent's active memory space. Because the configuration is held in volatile memory and checked against a remote cryptographic signature, local tampering becomes mathematically infeasible.
